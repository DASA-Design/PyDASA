## 2026-04-06 | reports - Independent codebase analysis

**Stage:** analysis

**Hypothesis:** Claude Code (Opus) can produce useful architecture, code review, and documentation reports independently, comparable to reports generated by other models.

**What was done:**
- Generated 3 independent reports from code only (did NOT read existing reports):
  1. `code_review_full_2026-04-06.md` — 7-section code review (abstractions, deps, risks, bugs, debt)
  2. `architecture_report_2026-04-06.md` — ADD-style architecture (13 FRs, 19 ASRs, 6 principles)
  3. `documentation_report_2026-04-06.md` — Documentation quality assessment (B+ grade)

**Key findings from code review:**
- 3 confirmed bugs (FDU set ordering, Dimension roundtrip, inverted conditional)
- God objects: Matrix (900+ lines), MonteCarlo (700+), Sensitivity (800+)
- Duplicated to_dict/from_dict across 4 classes
- 3 stub modules in context/ (entirely TODO)
- Strong patterns: decorator validation, JSON config singleton, compositional Variable design

**Key findings from architecture:**
- Schema object is the core architectural decision — parameterizes entire pipeline
- 6 design principles identified from code patterns
- CI/CD pipeline well-structured for single-maintainer workflow

**Key findings from documentation:**
- 99.8% docstring coverage, consistent Google style
- 10 empty __init__.py files, 1 published stub page, wrong module name in parser.py

**Next step:** Compare with third-party model reports. Analysis will check alignments and inconsistencies between the two sets.

**Decision:** accepted — reports are accurate with real issues and insights. Prose needs improvement (addressed in skill update). Skills are now calibrated for future analysis and state reports.

## 2026-04-06 | reports - Comparison with third-party model

**Stage:** analysis

**What was done:**
- Third-party reports (3 files) placed in `notes/reports/third-party/`
- Comparison analysis generated: `notes/reports/comparison_analysis_2026-04-06.md`

**Result:**
- Opus found 3 real bugs the third-party missed (FDU ordering, Dimension roundtrip, dead logic)
- Opus was more accurate on project metadata (deps, Python version, schema FDUs)
- Opus had 1 false positive (scipy.stats flagged as unused but used 750 lines below import)
- Third-party had better prose, academic presentation, and diagram descriptions
- Third-party had factual errors on basics (2 deps vs 7, Python 3.8 vs 3.10, 3 FDUs vs 7)

**Feedback received:**
- Opus reports are too hard to read — tables and facts are good but need clearer prose and insights
- Reports should end with proposals and ideas, not just diagnosis
- Third-party's narrative style is better for comprehension even if less accurate

**Decision:** accepted as baseline. Reports skill needs updating: add prose summaries per section, end with actionable proposals. Future reports should balance accuracy (Opus strength) with readability (third-party strength).
